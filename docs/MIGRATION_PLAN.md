# План миграции COLDDEV

## Аудит исходного репозитория

Исходный проект был Next.js App Router со статическим экспортом (`output: "export"`) и GitHub Pages workflow, который публикует папку `out` в `gh-pages`. Кастомный домен `colddev.pro` уже закреплён через `public/CNAME` и сохраняется.

Предыдущий слой был ретро-терминалом YAROSLAV.EXE: компоненты рабочего стола, boot-screen, окна Windows 2000 и контент портфолио. В нём не было маршрутов кабинета, API-контракта, сессий, данных проектов или админки. Поэтому оставлять старый UI и наращивать его было бы дороже и сложнее, чем заменить продуктовый слой, сохранив стек и деплой.

## Целевая архитектура

```text
colddev.pro (GitHub Pages / Next static export)
              ↓ POST text/plain JSON
Google Apps Script Web App
              ↓
Google Sheets (метаданные) + Google Drive (чеки и файлы)
```

Frontend не содержит ID таблицы, ключей или паролей. Без `NEXT_PUBLIC_APPS_SCRIPT_URL` используется локальный demo-адаптер; после добавления переменных в GitHub Repository Variables сборка переключается на Apps Script.

## Этапы

1. **Выполнено — основа и дизайн:** новый русский лендинг, общая визуальная система COLDDEV, адаптивная сетка и GitHub Pages-compatible static export.
2. **Выполнено — кабинет:** `/login`, `/demo`, `/dashboard`, переключатель нескольких проектов, этапы, обновления, сайт, рекламные отчёты, ЕРИП, счета, загрузка чеков и дополнительные услуги.
3. **Выполнено — админский UI:** `/admin`, Google Identity Services, разделы клиентов, проектов, этапов, обновлений, рекламы, счетов, услуг, портфолио и кейсов, формы создания/редактирования.
4. **Выполнено — backend scaffold:** `apps-script/Code.gs`, инициализация листов, хэширование кодов, сессии, Google token verification, Drive upload и API mutations.
5. **Следующий шаг — подключение:** создать Google Sheet, выполнить `setupColddev`, опубликовать Apps Script web app, заполнить GitHub variables и заменить demo seed-данные реальными строками таблиц.
6. **После подключения — hardening:** протестировать CORS/redirect Apps Script на домене, выключить `DEMO_MODE`, проверить OAuth origins, включить регулярную резервную копию таблицы и добавить rate limiting при росте нагрузки.

## MVP-ограничения

- Demo-данные нужны только для презентации и локальной разработки.
- Статистика Яндекс Директа на первом этапе вводится вручную в админке.
- Файл чека становится доступен backend-владельцу в Drive; клиент видит только статус обработки.
- Google Sheets + Apps Script подходят для небольшого числа клиентов, но не являются заменой промышленной БД при тысячах одновременных запросов.
